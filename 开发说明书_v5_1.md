# **1\. 系统概述**

## **1.1 系统定位**

本系统是一套智能楼宇设备管理后端服务，负责管理楼宇内断路器、温湿度传感器、电表等终端设备。系统通过 MQTT 协议与智能网关通信，实现设备控制指令下发和实时数据采集，并通过 REST API 和 WebSocket 向前端提供数据服务。

## **1.2 技术选型**

### **编程语言：Go**

Go（又称 Golang）是 Google 开发的静态类型、编译型语言，具备以下特点：

- 原生支持高并发（goroutine/channel），适合处理 MQTT 消息流和 WebSocket 连接池
- 单二进制部署，无需安装运行时环境，Windows/Linux 均可直接运行
- 标准库丰富，HTTP、JSON、数据库驱动开箱即用

### **Web 框架：Gin**

Gin 是 Go 生态最流行的 HTTP 框架，性能极高（基于 httprouter），提供路由分组、参数绑定、中间件等功能。本系统用 Gin 注册所有 REST API 路由和 WebSocket 端点。

### **数据库：SQLite（modernc/sqlite，纯 Go 实现）**

SQLite 是一款嵌入式关系型数据库，所有数据存储在单个 .db 文件中，无需独立数据库服务进程。本系统使用 modernc.org/sqlite 驱动，该驱动为纯 Go 实现，无需安装 GCC 或 CGO 工具链，可在任意平台直接编译运行，不依赖任何系统级 C 库。

- 适合边缘设备/网关服务器场景，无需额外运维数据库
- 数据量级（百台设备、秒级采集）完全在 SQLite 性能范围内
- 单文件备份简单，gateway.db 直接拷贝即可

### **MQTT 客户端：paho.mqtt.golang**

Eclipse Paho 是 MQTT 协议的官方参考实现，支持 MQTT 3.1/3.1.1，提供自动重连、QoS 0/1/2、遗嘱消息等功能。本系统使用 QoS 1（至少一次送达），确保控制指令和数据上报不丢失。QoS 级别可在 config.json 中配置。

### **WebSocket：gorilla/websocket**

gorilla/websocket 是 Go 生态最广泛使用的 WebSocket 库，符合 RFC 6455 标准。本系统用其实现设备实时数据推送，每个连接独立加写锁防止并发写 panic。

## **1.3 整体架构**

系统由五层构成，数据在各层之间按固定方向流动：

| **层级** | **组件** | **职责** |
| --- | --- | --- |
| 设备层 | 断路器 / 温湿度传感器 / 电表 | 执行控制指令，采集物理量数据 |
| 网关层 | 艾络科技 AN-G2i | RTU → MQTT 协议转换，IO 点位映射，串口通道管理 |
| 通信层 | MQTT Broker（端口 1883） | 消息路由中间件，不做任何业务处理 |
| 服务层 | Go 后端（本系统） | 设备管理、指令下发、数据采集存储、REST API、WebSocket 实时推送 |
| 存储层 | SQLite（gateway.db） | 持久化设备静态信息（devices）、实时数据（breaker_data 及各设备类型动态数据表）、MQTT 配置历史（mqtt_configs），单文件存储，无需独立数据库服务 |

## **1.4 数据流向总览**

### **控制流（下行）**

- 前端调用 POST /control 接口，传入 action 参数
- 服务端查询数据库，取得该设备的 channel 和 gateway
- 服务端拼接 ctrlPnt 路径，构造 MQTT 控制报文
- 发布到 MQTT Topic：webScadaCtrl
- 网关订阅该 Topic，收到报文后操作对应断路器

### **数据流（上行）**

- 网关采集设备数据，按配置的上报频率主动推送
- 发布到 MQTT Topic：webScadaCtrl，报文格式为 dataType=realtime
- 服务端订阅该 Topic，收到报文后解析
- 按设备类型写入对应数据库状态表
- 通过 WebSocket 广播 update 消息给所有已连接的前端

## **1.5 设备类型**

| **device_type** | **名称** | **是否可控制** | **数据点** |
| --- | --- | --- | --- |
| breaker | 智能断路器 | ✅ 支持 | breaker_work / voltage_a / current_a / power_a / frequency |
| env_sensor | 温湿度传感器 | ❌ 不支持 | temperature / humidity |
| meter | 智能电表 | ❌ 不支持 | meter |

_⚠ 只有 device_type=breaker 的设备允许调用 /control 接口，其他类型调用返回 400。_

## **1.6 动静分离**

系统将设备数据明确分为两类，采用完全不同的存储和获取方式：

| **类别** | **定义** | **存储位置** | **变化频率** | **获取方式** |
| --- | --- | --- | --- | --- |
| 静态数据（info） | 设备管理信息，由人工录入和维护 | devices 表 | 人工修改才变 | GET /info 或 WS snapshot 的 info 字段 |
| 动态数据（data） | 设备实时运行数据，由网关上报 | breaker_data 及各设备类型动态数据表（{device_type}\_data） | 随设备状态实时变化 | GET /data 或 WS snapshot 的 data 字段 + update 消息 |

_ℹ 前端主页面只需连接 WebSocket，服务端自动推送完整数据（info+data），无需单独调用 REST 接口。_

## **1.7 三语义统一**

所有接口、WebSocket 消息、数据库设计均围绕三个语义方向统一设计：

| **语义** | **方向** | **含义** | **REST 接口** | **WS 消息** |
| --- | --- | --- | --- | --- |
| info | 静态  | 设备管理信息（名称、位置、管理员等） | GET /info | snapshot.info |
| data | 上行（采集） | 设备所有动态数据（状态码+物理量数值） | GET /data | snapshot.data + update |
| control | 下行（控制） | 控制指令下发（仅断路器） | POST /control | —（仅下行，无推送） |

## **1.8 设备识别号规范**

每台终端设备有唯一识别号，格式如下：

格式：B{楼栋}\_R{房间}\_T{序号}

示例：B1_R101_T01 → 1号楼 101室 第1台

示例：B2_R305_T03 → 2号楼 305室 第3台

_⚠ 终端识别号同时用于 MQTT ctrlPnt 路径和数据库主键，网关后台 IO 点位配置必须与本规范完全一致，包括下划线格式。_

# **2\. 术语表**

| **术语** | **全称/说明** |
| --- | --- |
| MQTT | Message Queuing Telemetry Transport，轻量级发布/订阅消息协议，专为低带宽、不稳定网络设计，标准端口 1883 |
| RTU | Remote Terminal Unit，远程终端单元，通过串口（RS485/RS232）与传感器/断路器通信的硬件设备 |
| Modbus RTU | 基于串口的工业通信协议，断路器通过此协议暴露寄存器供网关读写 |
| QoS | Quality of Service，MQTT 消息服务质量等级。0=最多一次（可能丢），1=至少一次（可能重复），2=恰好一次 |
| Topic | MQTT 消息主题，发布者和订阅者通过 Topic 路由消息，本系统使用 webScadaCtrl |
| Broker | MQTT 消息代理服务器，负责接收和转发消息，本系统对接端口 1883 |
| SQLite | 嵌入式关系型数据库，数据存储在单个文件中，无需独立数据库服务进程 |
| modernc/sqlite | 纯 Go 实现的 SQLite 驱动，无需 GCC/CGO，可在 Windows/Linux 直接编译 |
| Gin | Go 语言 HTTP Web 框架，高性能路由，本系统用于注册所有 REST API |
| gorilla/websocket | Go 语言 WebSocket 库，符合 RFC 6455 标准，本系统用于实时推送设备数据 |
| ctrlPnt | Control Point，MQTT 控制报文中标识目标寄存器的路径字符串，格式：{协议}:db.{channel}.{gateway}.breaker_control |
| channel | 网关串口通道编号，如 c1/c2，标识断路器接在网关哪个串口上 |
| gateway | 终端识别号，格式 B{楼栋}\_R{房间}\_T{序号}，用于唯一标识一台终端设备 |
| snapshot | WebSocket 消息类型，新连接时推送设备完整数据（info + data） |
| update | WebSocket 消息类型，数据变化时推送单字段增量更新 |
| breaker_work | 断路器分合闸工作状态码，整数 0-16，含义见 3.3 节 |
| breaker_control | 断路器控制寄存器 tag 名，写入 targetValue 触发分合闸动作 |
| targetValue | 控制指令值，由 action 字符串映射而来，写入 MQTT 控制报文 |
| 倍率  | 网关 IO 点位配置中的数值换算系数，将寄存器原始值转换为工程单位值 |
| info | 三语义之一，指设备静态管理信息，存 devices 表 |
| data | 三语义之一，指设备动态运行数据，存 \*\_data 表 |
| control | 三语义之一，指控制指令下发，不存储，直接转发至 MQTT |

## **3.7 新增设备类型操作说明**

v3 起，后端对设备数据处理采用动态驱动架构，breaker 保留强类型处理（支持控制指令），其他设备类型统一走动态路径，新增任何设备类型只需在数据库层操作，代码无需改动。

### **核心约定**

| **约定项** | **规则** |
| --- | --- |
| 数据表命名 | {device_type}\_data，device_type 与 devices 表中的 device_type 字段值完全一致 |
| 字段命名 | 数据字段名即网关 IO 点位配置的 tag 名，大小写敏感，必须完全一致 |
| 必要字段 | gateway TEXT PRIMARY KEY、updated_at DATETIME DEFAULT CURRENT_TIMESTAMP |
| 数据类型 | 数值型字段使用 REAL，状态码类整数字段使用 INTEGER |

### **第一步：建数据表**

根据设备实际采集的数据点建表，每个数据点对应一个字段：

CREATE TABLE {device_type}\_data (

gateway TEXT PRIMARY KEY,

{tag_1} REAL, -- 数据点1，字段名 = 网关 tag 名

{tag_2} REAL, -- 数据点2，字段名 = 网关 tag 名

\-- 根据实际数据点继续添加...

updated_at DATETIME DEFAULT CURRENT_TIMESTAMP

);

### **第二步：插入设备记录**

往 devices 表插入该设备，device_type 填写与第一步表名前缀完全一致的值：

INSERT INTO devices (id, building, room, gateway, channel, device_type, name, location, admin, phone)

VALUES ('{id}', '{building}', '{room}', '{gateway}', '{channel}', '{device_type}', '{name}', '{location}', '{admin}', '{phone}');

### **第三步：网关 IO 点位配置**

在网关后台为该设备配置所有数据点的 tag，tag 名称必须与第一步建表的字段名完全一致：

| **配置项** | **要求** |
| --- | --- |
| tag 名称 | 与数据库表字段名完全一致，区分大小写 |
| 数据类型 | 数值型选 FLOAT，整数状态码选 INT16/UINT16 |
| 方向  | 数据采集均为读（上行），控制寄存器为写（下行，仅 breaker 类型支持） |
| 倍率  | 按硬件手册配置，上报值 = 寄存器原始值 × 倍率 |

_⚠ tag 名与表字段名不一致时，UPDATE 影响 0 行，数据静默丢失不报错。建表和网关配置必须同步完成后再上线设备。_

### **完成后后端自动处理的流程**

- 网关上报报文到达 → 按 dev 字段解析出 gateway
- 查 devices 表得到该设备的 device_type
- 拼数据表名：{device_type}\_data
- 解析报文所有 tag，合并为一条 SQL 写入：INSERT OR IGNORE 确保行存在，再 UPDATE {device_type}\_data SET tag1=?,tag2=?,... WHERE gateway=?（一条报文一次写库）
- 写库成功后广播一次 WebSocket update 消息，fields 字段包含本次报文所有变化的 tag 和新值
- GET /{building}/{room}/{gateway}/data 执行 SELECT \* FROM {device_type}\_data WHERE gateway=?，返回该设备所有字段

### **示例：新增水压传感器**

以下为完整操作示例，device_type 定为 water_pressure，采集两个数据点：pressure（水压，单位 MPa）和 flow_rate（流量，单位 m³/h）。

第一步 — 建表：

CREATE TABLE water_pressure_data (

gateway TEXT PRIMARY KEY,

pressure REAL,

flow_rate REAL,

updated_at DATETIME DEFAULT CURRENT_TIMESTAMP

);

第二步 — 插入设备记录：

INSERT INTO devices (id, building, room, gateway, channel, device_type, name, location, admin, phone)

VALUES ('10104', '1', '101', 'B1_R101_T04', 'c3', 'water_pressure', '101室水压传感器', '管道间', '张三', '13800000001');

第三步 — 网关 IO 点位配置（tag 名与表字段名完全一致）：

| **tag 名** | **数据类型** | **倍率** | **单位** | **方向** |
| --- | --- | --- | --- | --- |
| pressure | FLOAT | ×0.01 | MPa | 读（上行） |
| flow_rate | FLOAT | ×0.1 | m³/h | 读（上行） |

网关上报报文示例：

{

"dataType": "realtime",

"time": "2026-03-11 10:00:00",

"datas": \[{

"dev": "c3.B1_R101_T04",

"data": { "pressure": "1.25", "flow_rate": "3.80" }

}\]

}

GET /1/101/B1_R101_T04/data 响应：

{

"gateway": "B1_R101_T04",

"pressure": 1.25,

"flow_rate": 3.80,

"updated_at": "2026-03-11 10:00:00"

}

# **4\. IO 点位 Tag 命名规范**

## **4.1 设计原则**

- Tag 名称全部小写，多词用下划线连接
- Tag 是该设备在网关中对应寄存器的物理标识，以设备名（终端识别号）作为命名空间
- 网关后台 IO 点位配置的 tag 名称必须与本规范完全一致，区分大小写
- breaker_control 是唯一的写方向 tag，其余均为读方向
- 物理量命名采用「物理量_相位」格式（如 voltage_a），预留三相扩展空间

## **4.2 完整 Tag 清单**

| **tag 名称** | **语义分类** | **适用设备** | **方向** | **数据类型** | **倍率** | **单位** | **说明** |
| --- | --- | --- | --- | --- | --- | --- | --- |
| breaker_control | control | 断路器 | 写（下行） | INT16 | ×1  | —   | 控制指令寄存器，写入 targetValue 触发动作 |
| breaker_work | data | 断路器 | 读（上行） | UINT16 | ×1  | —   | 分合闸工作状态码，整数 0-16 |
| voltage_a | data | 断路器 | 读（上行） | UINT16 | ×0.1 | V   | A 相电压，原始单位 0.1V |
| current_a | data | 断路器 | 读（上行） | UINT16 | ×0.01 | A   | A 相电流，原始单位 0.01A |
| power_a | data | 断路器 | 读（上行） | INT16 | ×10 | W   | A 相有功功率，原始单位 0.01kW，×10 转 W |
| frequency | data | 断路器 | 读（上行） | UINT16 | ×0.01 | Hz  | 电网频率，原始单位 0.01Hz |
| temperature | data | 温湿度传感器 | 读（上行） | FLOAT | ×1  | °C  | 环境温度 |
| humidity | data | 温湿度传感器 | 读（上行） | FLOAT | ×1  | %RH | 环境湿度 |
| meter | data | 电表  | 读（上行） | FLOAT | ×1  | kWh | 累计用电量 |

_⚠ power_a 原始寄存器单位为 0.01kW（INT16 有符号），网关倍率配 ×10 后上报单位为 W。数据类型必须配 INT16（有符号），否则负功率值会溢出为大正数。_

## **4.3 断路器 breaker_work 状态码对照表**

固件内置编码，服务端原样存储，不做额外转换，前端按需展示：

| **状态码** | **含义** | **状态码** | **含义** |
| --- | --- | --- | --- |
| 0   | 手动分闸 | 9   | 机构挂锁 |
| 1   | 指令分闸 | 10  | 定时分闸 |
| 2   | 故障分闸 | 11  | 定时合闸 |
| 3   | （保留） | 12  | （保留） |
| 4   | 手动合闸 | 13  | 干接点分闸 |
| 5   | 指令合闸 | 14  | 干接点合闸 |
| 6   | 自动合闸 | 15  | 手动模式分闸 |
| 7   | 指令锁死（需解锁） | 16  | 手动模式合闸 |
| 8   | 故障锁死（需解锁） | —   | —   |

## **4.4 控制动作与 targetValue 映射**

前端调用 /control 接口时传入 action 字符串，服务端自动映射为对应 targetValue 写入 MQTT 报文。targetValue 来源于断路器 Modbus 协议寄存器 6802 定义（十六进制转十进制）：

| **action（前端传入）** | **targetValue（MQTT报文）** | **十六进制原值** | **说明** |
| --- | --- | --- | --- |
| close | 51  | 0x33 | 合闸  |
| open | 19  | 0x13 | 分闸不锁死 |
| lock | 3   | 0x03 | 分闸锁死（锁死后必须先 unlock 才能合闸） |
| unlock | 35  | 0x23 | 分闸解锁 |

## **4.5 采集周期建议**

_⚠ 建议网关采集周期设置为 5 秒或以上。RTU 串口为半双工通信，采集（读）和控制（写）共用同一总线，采集频率过高会导致控制指令被延迟或丢弃，表现为控制命令概率性无响应。经实测，5 秒周期可有效避免总线竞争问题。_

# **5\. MQTT 通信规范**

## **5.1 连接配置**

| **参数** | **值** | **说明** |
| --- | --- | --- |
| Topic | webScadaCtrl | 控制下发与数据上报共用同一 Topic，可在 config.json 修改 |
| QoS | 1（可配置） | 至少一次送达，确保控制指令和数据不丢失，在 config.json 的 qos 字段配置 |
| ClientID | smart-gateway-{时间戳} | 每次启动自动生成，避免冲突 |
| AutoReconnect | true | 断线自动重连 |

## **5.2 Topic 设计**

控制下发与状态上报使用同一个 Topic，服务端通过报文内字段区分方向：

| **Topic** | **方向** | **区分方式** |
| --- | --- | --- |
| webScadaCtrl | 服务端 → 网关（控制下发） | 报文包含 msgType=CTRL_REQ |
| webScadaCtrl | 网关 → 服务端（数据上报） | 报文包含 dataType=realtime |

_⚠ 服务端订阅 webScadaCtrl 后会收到自己发出的控制报文，必须检查 dataType 字段，只处理 realtime 类型，过滤掉 CTRL_REQ。_

## **5.3 控制下发报文格式**

服务端发布到 webScadaCtrl 的控制报文结构：

| **字段** | **示例值** | **说明** |
| --- | --- | --- |
| ctrlProcId | Webscada_1718902666174_1234 | 唯一流水号，格式：Webscada_{毫秒时间戳}\_{随机数} |
| msgType | CTRL_REQ | 固定值 |
| time | 1718902666174 | Unix 毫秒时间戳 |
| msgFrom | Webscada | 固定值 |
| protocol | dc_gm_rtu | 来自 config.json protocol 字段 |
| ctrlPnt | {dc_gm_rtu1}:db.c1.B1_R101_T01.breaker_control | 动态拼接，见下方规则 |
| targetValue | 51  | 控制动作值，见 3.4 节映射表 |
| isTest | false | 固定 false |

### **ctrlPnt 路径拼接规则**

格式：{dc_gm_rtu1}:db.{channel}.{gateway}.breaker_control

示例：{dc_gm_rtu1}:db.c1.B1_R101_T01.breaker_control

| **组成部分** | **来源** | **示例值** |
| --- | --- | --- |
| {dc_gm_rtu1} | config.json 中 protocol 字段拼接 1 | 固定为 dc_gm_rtu1 |
| channel | devices 表 channel 字段 | c1  |
| gateway | devices 表 gateway 字段（终端识别号） | B1_R101_T01 |
| breaker_control | 固定 tag 名称 | 固定值 |

### **完整报文示例**

{

"ctrlProcId": "Webscada_1741234567890_8821",

"msgType": "CTRL_REQ",

"time": 1741234567890,

"msgFrom": "Webscada",

"protocol": "dc_gm_rtu",

"ctrlPnt": "{dc_gm_rtu1}:db.c1.B1_R101_T01.breaker_control",

"targetValue": 51,

"isTest": false

}

## **5.4 网关上报报文格式**

网关 mqtt-common 上行通道推送的真实报文结构（已实测确认）：

| **字段** | **示例值** | **说明** |
| --- | --- | --- |
| dataType | realtime | 固定值，服务端靠此字段过滤 |
| time | 2026-03-09 15:30:44 | 上报时间，字符串格式 |
| datas | \[...\] | 数组，一条报文可包含多台设备 |
| datas\[\].dev | c1.B1_R101_T01 | 格式：{channel}.{gateway}，按第一个 . 分割 |

## **5.5 服务端 MQTT 处理逻辑**

服务端收到 webScadaCtrl Topic 的每条消息后，执行以下流程：

- 检查报文是否包含 dataType=realtime，不包含则直接丢弃（过滤自己发出的控制报文）
- 解析 JSON，遍历 datas 数组
- 取 dev 字段，按第一个 . 分割，得到 channel 和 gateway
- 用 gateway 查询 devices 表，取得 device_type
- 按 device_type 分发：breaker → handleBreakerData（强类型处理）；其他所有类型 → handleDynamicData（动态写库，无需改代码）
- 解析报文所有 tag，ParseFloat 转换后合并为一条 SQL 写入：INSERT OR IGNORE 确保行存在，再 UPDATE {device_type}\_data SET tag1=?,tag2=?,... WHERE gateway=?（一条报文一次写库，解析/拼SQL/填fields在同一次循环完成，不存在错位）
- 写库成功后一次广播 WSUpdate 消息，fields 包含本次报文所有变化的 tag 和新值

| db/db.go | 数据库初始化与全部 CRUD 操作。包含：建表（5张表）、设备增删改查（devices）、断路器合并写库（UpsertBreakerAll：强类型，解析/拼SQL/填fields一次循环完成）、其他设备类型动态合并写库（UpsertFieldBatch：动态拼表名字段名，解析/拼SQL/填fields一次循环完成）、动态查询（GetDeviceData：SELECT \* FROM {device_type}\_data）、MQTT 配置读写（mqtt_configs）、联合查询 GetFullSnapshot（供 WebSocket snapshot 使用）。 |
| db/seed.go | 示例数据初始化。首次启动时向 devices / breaker_data / env_sensor_data / meter_data 写入演示用数据，便于开发调试，不影响生产运行。新增设备类型的示例数据需自行建表后手动插入。 |
| model/model.go | 全局数据模型定义。包含所有结构体：Device（设备静态信息）、BreakerData（断路器实时数据，强类型）、MQTTControlPayload（控制下发报文）、MQTTUploadPayload / MQTTDeviceData（网关上报报文）、ControlRequest / ControlResponse（控制接口请求响应）、WSSnapshot / WSUpdate（WebSocket 消息，breaker 的 data 字段为强类型 BreakerData，其他设备类型 data 字段为 map\[string\]interface{}）、MQTTConfig（MQTT 配置历史）、ActionValueMap（action → targetValue 映射表）。 |
| mqtt/mqtt.go | MQTT 客户端全部逻辑。包含：Connect / Disconnect / IsConnected / Status（连接管理，Status 返回连接状态和当前 Broker 地址）、subscribe / handleMessage（订阅与报文分发）、handleBreakerData（断路器强类型合并写库）、handleDynamicData（其他所有设备类型动态合并写库：动态拼表名字段名 → 一条SQL写入 → 一次广播 WebSocket）、PublishControl（构造并发布控制指令报文）、InitSnapshotCallback（注入 WebSocket 新连接时的 snapshot 推送回调，避免循环依赖）。 |
| ws/hub.go | WebSocket 连接池管理。Hub 结构体维护所有活跃连接（map），ServeWS 升级 HTTP 连接为 WebSocket 并触发 OnConnect 回调推送 snapshot，Broadcast 向全部连接广播消息（每连接独立 goroutine 发送，互不阻塞），client 结构体内置写锁防止并发写 panic。 |
| config.json | 运行时配置文件。字段：port（HTTP 端口）、db_path（数据库文件路径）、protocol（MQTT ctrlPnt 协议前缀）、topic（MQTT Topic 名称）、qos（MQTT 服务质量等级）。首次启动自动生成，可直接编辑后重启生效。 |
| gateway.db | SQLite 数据库文件。包含 5 张表的全部持久化数据。直接拷贝此文件即可完成备份或迁移。 |
| go.mod | Go 模块定义文件。声明模块名（smart-gateway）和 Go 版本，列出直接依赖：gin / paho.mqtt.golang / gorilla/websocket / modernc.org/sqlite。 |
| go.sum | 依赖校验文件。由 go mod tidy 自动生成，记录所有依赖包的哈希值，确保构建可重现，不需手动修改。 |

# **9\. 服务启动与配置**

## **9.1 配置文件 config.json**

服务首次启动时自动生成，可手动修改：

| **字段** | **默认值** | **说明** |
| --- | --- | --- |
| port | :8080 | HTTP 监听端口 |
| db_path | gateway.db | SQLite 数据库文件路径 |
| protocol | dc_gm_rtu | MQTT ctrlPnt 协议前缀 |
| topic | webScadaCtrl | MQTT Topic 名称，控制下发和数据上报共用 |
| qos | 1   | MQTT QoS 等级：0=最多一次，1=至少一次，2=恰好一次 |

完整 config.json 示例：

{

"port": ":8080",

"db_path": "gateway.db",

"protocol": "dc_gm_rtu",

"topic": "webScadaCtrl",

"qos": 1

}

## **9.4 部署注意事项**

| **事项** | **说明** |
| --- | --- |
| 防火墙端口 | 服务监听 8080（HTTP/WebSocket），MQTT Broker 使用 1883。部署前确认这两个端口在防火墙/安全组中已放行，否则前端无法连接。 |
| SQLite 文件权限 | gateway.db 所在目录需要进程读写权限。Linux 下建议确认文件 owner 与运行用户一致，避免 permission denied 启动失败。 |
| gateway.db 备份 | 直接拷贝 gateway.db 文件即可完成备份。建议定期备份，服务停止后再拷贝确保完整性。 |
| 日志与监控 | 当前版本输出到 stdout，建议用 systemd 或 nohup 重定向到日志文件：./smart-gateway >> app.log 2>&1 & |
| Windows 服务注册 | 如需开机自启，可使用 NSSM（Non-Sucking Service Manager）将可执行文件注册为 Windows 服务。 |
| Linux 服务注册 | 推荐使用 systemd，创建 /etc/systemd/system/smart-gateway.service 单元文件，配置 Restart=always 实现自动重启。 |
| MQTT 连接失败 | 服务启动时 MQTT 连接失败不影响 HTTP 服务运行，可通过 POST /mqtt/connect 接口在运行时重连，无需重启服务。 |
| config.json 修改 | 修改 config.json 后需重启服务才能生效。port 和 db_path 修改需特别注意，确保新路径有写权限。 |

## **9.5 MQTT 启动向导**

服务启动时自动进入 MQTT 连接向导，流程如下：

- 检查数据库历史配置
- 有历史配置：显示上次 Broker 地址，倒计时 10 秒自动连接；可输入 new 切换、list 查看历史
- 无历史配置：引导输入 Broker 地址、用户名、密码（回车跳过）
- 连接失败不影响 HTTP 服务启动，可通过 POST /mqtt/connect 重连
